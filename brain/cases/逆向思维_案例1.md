# 案例：逆向思维 — 如何快速定位 Bug

> **方法**：逆向思维
> **适用场景**：从目标反推路径，从结果反推原因

---

## 问题背景

用户问："登录一直失败，但后端日志没有任何错误，怎么排查？"

正向思维（试错）：
- 检查前端控制台 → 没看到明显报错
- 检查网络请求 → 状态码200
- 检查数据库 → 用户存在
- 继续试错... 30分钟后还是没找到

---

## 逆向思维执行

### Step 1: 定义目标（结果）

> "登录成功"是什么样？
> - 前端：显示登录成功，跳转首页
> - 后端：返回 token，用户信息
> - 数据库：last_login 更新

### Step 2: 从目标倒推

```
登录成功 ← 前端收到 200 + token
        ← 后端验证通过
        ← 数据库查询到用户 + 密码匹配
```

### Step 3: 从失败状态反推

```
登录失败 ← 哪里断裂了？

断裂点1：前端没收到200？
  → 检查网络，实际：收到了（状态码200）

断裂点2：后端没验证通过？
  → 检查业务逻辑
  → 实际：代码里有个"账号锁定"逻辑
  → 被锁定的用户返回200但不走正常流程

断裂点3：数据库查不到用户？
  → 概率低，但可以查

结论：后端"账号锁定"逻辑是罪魁祸首
```

### Step 4: 验证

检查账号锁定表 → 果然该用户被锁定

### Step 5: 解决

解锁账号 → 登录成功

---

## 关键洞察

| 思维模式 | 做法 | 耗时 |
|---------|------|------|
| 正向试错 | 逐一排查可能原因 | 30分钟+ |
| 逆向思维 | 从目标倒推，找到断裂点 | 5分钟 |

---

## 逆向思维检查清单

```
目标是什么？________________

成功的标志是什么？________________

从成功往回推：
  ← 需要什么？
    ← 需要什么？
      ← 需要什么？

现在断裂在哪一步？
________________
```

---

## 适用场景

- Bug 定位（日志没错误但功能不工作）
- 需求分析（不知道怎么做，从"什么才算成功"倒推）
- 决策困境（不知道该选哪个，从"我想要什么结果"反推）
