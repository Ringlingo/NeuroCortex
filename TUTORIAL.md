## 对比实验：没有框架 vs 有框架

### 场景

用户问了一个需要"持续上下文"的问题：

> **"我的小说写到第三章了，林羽在修炼第一境时遇到了瓶颈，怎么设计接下来的冲突比较好？"**

---

## ❌ 没有 NeuroCortex（普通 AI 对话）

```
用户：我的小说写到第三章了，林羽在修炼第一境时遇到了瓶颈，怎么设计接下来的冲突比较好？

AI：这个问题很有意思！以下是一些建议：
1. 可以让林羽遇到一个更强的对手...
2. 或者让他陷入内心的挣扎...
3. 也可以加入一个神秘人物...

（用户需要重新解释：这是修仙世界观、修炼体系、主角设定等）
```

**问题**：
- ❌ AI 不知道你的世界观
- ❌ 每次都要重新解释背景
- ❌ 回答泛泛而谈，没有针对性

---

## ✅ 有 NeuroCortex（加载框架后）

```
用户：我的小说写到第三章了，林羽在修炼第一境时遇到了瓶颈，怎么设计接下来的冲突比较好？

🧠 NeuroCortex 已激活
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

[加载域] writing (priority=7, context_boost=3)
[触发直觉] INT-003: 敏感内容 → 确保100%虚拟化
[加载知识] novels/序元纪年/CHARACTERS/林羽.md
[加载知识] novels/序元纪年/CULTIVATION_SYSTEM.md

━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

**1. 内部冲突（推荐）**
林羽的瓶颈源于"心魔"——他对父亲失踪的愧疚与修炼目标产生了潜意识冲突。
这与角色设定中"林羽的父亲林建国是量子生物学家"形成呼应。

**2. 外部冲突**

- ✅ 主动触发直觉规则（确保虚拟化）

## 第二个场景：域冲突裁决演示

### 用户问

> "这个 SQL 查询太慢了，怎么优化？"

---

## ❌ 没有域冲突裁决（普通 AI）

```
AI：这个SQL慢的原因可能是：
1. 没有建索引
2. 数据量太大
3. 查询语句写得不优

建议加索引、限制返回条数、优化JOIN...
```

**问题**：
- 没有利用你项目已有的数据库规范
- 可能给出不适用于你项目的建议

---

## ✅ 有域冲突裁决（智能加载）

```
用户：这个 SQL 查询太慢，怎么优化？

━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
🧠 NeuroCortex 启动序列
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

[Step 1] 安全检查 ✅
[Step 2] 注意力扫描 → 匹配域：coding(8) + data(7)
         ↓
         域冲突检测！执行三级裁决：
         ① priority：coding(8) > data(7)
         ② 上下文窗口：无上一轮，skip
         ③ 裁决结果：加载 coding 域

[Step 3] 跨域引用 → @ref:data/performance
[Step 4] INTUITION 扫描 → INT-002 触发（高危操作）⚠️
         → 先扫描再报告，风险提示

━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

AI：在优化之前，先确认几点：

⚠️ **高危操作提醒**：修改查询前建议备份

**根据 coding 域的项目规范**（@ref:data/performance）：

1. **先看执行计划**（你用的是 PostgreSQL？）
   EXPLAIN ANALYZE SELECT ...;

2. **根据计划判断**：
   - 如果 `Seq Scan` 多 → 考虑加索引
   - 如果 `Hash Join` 慢 → 检查统计信息

3. **你项目里的规范**：
   - 超过100ms的查询需要记录
   - 禁止全表扫描的 WHERE 条件已在 data/performance 中定义

需要我帮你分析具体的执行计划吗？
```

**效果**：
- ✅ 自动识别高危操作，触发风险提示
- ✅ 加载了你项目的数据库规范
- ✅ 给出针对性的优化建议
